## 1. Purpose

This document records material revisions across the NervLock ecosystem.

It exists to preserve:

- What changed
- Why it changed
- Who authorized the change
- Which version was replaced
- Which version became active
- Which documents, pages, channels, products, and derivative assets were affected
- Whether re-review was required
- Whether rollback was possible
- Whether the change was fully propagated
- Whether old versions were archived
- Whether the revision introduced residual risk

This revision log is not a list of every edit.

It is a controlled history of changes that may affect:

- Strategy
- Medical meaning
- Scientific interpretation
- Regulatory status
- Product safety
- Public claims
- Website structure
- Search performance
- User experience
- Brand identity
- AI operating rules
- Publication approval
- Long-term trust

---

## 2. Governing Revision Principle

NervLock follows one central revision rule:

> A material change is not complete until the change, reason, approval, affected assets, propagation status, and active version are recorded.

A revision is not complete merely because:

- A file was overwritten
- Code was deployed
- A caption was edited
- A product formula changed
- A reference was replaced
- A page was unpublished
- An AI prompt was updated
- A correction was posted

The revision must remain traceable.

### Required Linkage

A material revision should reference the relevant Decision ID when the change resulted from a strategic decision.

A decision that causes no implementation or document change may appear only in W04.

## 5. What Does Not Need a Formal Revision Entry

Routine low-impact edits generally do not require an individual W05 entry.

Examples:

- Typographic correction that does not alter meaning
- Spacing
- Non-material punctuation
- Internal task-status update
- Routine scheduling change
- Exporting the same approved asset into another file type
- Reordering internal notes without changing authority
- Routine weekly calendar maintenance
- Non-material CSS alignment fix
- File transfer with no change in content or status

However, log the change when a minor edit:

- Alters medical meaning
- Changes legal meaning
- Changes a number
- Changes a dose
- Changes a date
- Changes a warning
- Changes an approval status
- Changes a URL
- Changes a reference
- Changes a public visual implication

## 7. Revision Severity

### Critical

A revision required to prevent or correct serious harm, unlawful publication, major deception, privacy breach, security breach, product-safety failure, or severe reputational damage.

Examples:

- Dangerous medical error
- Disease-treatment claim for a supplement
- Wrong product dose
- Missing serious warning
- Unconsented patient information
- Fabricated reference
- Product recall
- Compromised website

Required action:

- Stop use or distribution
- Escalate immediately
- Correct or withdraw
- Preserve evidence
- Complete CAPA
- Record all affected assets

### Major

A revision that materially changes meaning, compliance, safety, strategy, product, or technical behavior.

Examples:

- Evidence certainty downgraded
- Product formula changed
- Blog taxonomy changed
- Canonical structure changed
- Medical red flags added
- Regulatory classification changed
- Major website migration

Required action:

- New review
- New approval
- Propagation
- Archive old version
- Update affected derivatives

### Moderate

A meaningful improvement that does not fundamentally change the core conclusion or authority.

Examples:

- Expanded limitations
- Improved accessibility
- Updated internal links
- Revised structured data
- Clarified terminology
- Improved mobile layout

Required action:

- Appropriate review
- Record affected assets
- Re-test where relevant

### Minor

A low-impact change with no material effect on meaning, risk, regulation, safety, or strategy.

Examples:

- Typographic correction
- Spacing
- Non-material formatting

May be grouped into a periodic maintenance entry.

## 10. Versioning Standard

### 10.1 Major Version

Increase the first number when the change materially alters:

- Authority
- Strategy
- Scope
- Architecture
- Risk controls
- Approval system
- Product formula
- Public claim territory

Example:

`1.0 → 2.0`

### 10.2 Minor Version

Increase the second number when the change adds or revises important content without replacing the overall framework.

Example:

`1.0 → 1.1`

### 10.3 Patch Version

A third number may be used for minor non-material corrections when operationally useful.

Example:

`1.1 → 1.1.1`

Patch versions should not hide material changes.

### 10.4 Dynamic Working Documents

Dynamic documents may use:

- Version number
- Last Updated date
- Revision ID
- Current status

Frequent routine updates may be grouped, but material decisions and changes require individual entries.

## 11. File Naming Standard

The canonical file name should match the Document ID.

Examples:

- `16_QA_Checklist.md`
- `W05_Revision_Log.md`

Avoid active file names such as:

- `final.md`
- `final2.md`
- `latest.md`
- `new.md`
- `download.md`
- `QA_Checklist.md-다운로드.txt`

Downloaded or platform-generated suffixes may exist temporarily.

They must not be treated as the authoritative canonical name.

---

## 12. Source-of-Truth Revision Rule

When a document changes:

1. Confirm the active source
2. Compare the old and new version
3. Assign a Revision ID
4. Record the reason
5. Obtain required approval
6. Update version and date
7. Replace active copies in approved systems
8. Archive the old version
9. Check related documents
10. Check task prompts
11. Check implementation repositories
12. Check public assets
13. Verify synchronization

The most recently uploaded file is not automatically authoritative.

12. Complete W05 entry

## 35. Review Requirements by Severity

### Critical

Required:

- Immediate human escalation
- Medical review where relevant
- Regulatory review where relevant
- Legal or privacy review where relevant
- Technical or quality review
- Final human approval
- Public correction or notice assessment
- CAPA

### Major

Required:

- Relevant specialist review
- Final human approval
- Propagation
- Verification
- Archive

### Moderate

Required:

- Relevant owner review
- Appropriate QA
- Verification
- Log entry

### Minor

May be:

- Grouped
- Reviewed during routine maintenance
- Recorded as a batch entry

## 36. Revision Approval Boundary

Only an authorized human reviewer may approve:

- Core governance version change
- Medical conclusion change
- Regulatory interpretation change
- Product formula change
- Public claim change
- Website architecture change
- Product launch or withdrawal
- Critical correction
- Final rollback

AI may:

- Compare versions
- Identify changes
- Draft the revision entry
- Identify affected assets
- Propose propagation
- Check consistency
- Prepare rollback instructions

AI may not approve its own revision.

---

## 37. Revision Verification

Verification should confirm:

- Correct version is active
- Old version is archived
- Public behavior matches the revision
- Medical meaning is preserved
- Regulatory requirements are met
- References work
- Links work
- Technical tests pass
- Derivatives are addressed
- Search signals are correct
- Product information is consistent
- Customer-support materials are updated
- AI environments use the correct version

Verification should be completed by someone other than the sole implementer for high-risk changes when feasible.

## 42. Final Revision Principles

NervLock should revise when revision is necessary.

It should not hide change to appear consistent.

Consistency means applying the same standards.

It does not mean refusing to correct outdated conclusions.

A correction strengthens trust when it is:

- Prompt
- Accurate
- Visible
- Proportionate
- Fully propagated
- Properly recorded

A new upload is not automatically a new version.

A new version is not automatically approved.

An implemented change is not automatically verified.

A corrected source is not fully corrected until its material derivatives are corrected.

A product formula is not changed until label, safety, quality, content, support, inventory, and regulatory implications are addressed.

The revision log is complete only when another reviewer can determine:

- What changed
- Why
- Who approved it
- Where it was implemented
- Whether every affected asset was addressed
- Which version is now authoritative

NervLock must preserve both the current truth and the history of how it arrived there.
